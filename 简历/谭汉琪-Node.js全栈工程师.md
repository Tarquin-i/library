## 项目经验

### EzLottery - 彩票兑奖及库存管理系统
*个人项目 | 约 51,000 行代码*

**技术栈**：Next.js 15, React 19, Hono.js, Bun, PostgreSQL (ltree), Drizzle ORM, TanStack Query, shadcn/ui, Recharts, Tauri

**负责模块**：钱包模块、版本管理模块、成员树状显示模块

#### 钱包模块（余额管理、充值/提现申请、信用额度）

**前端**：
- 使用 React 19 + TanStack Query 构建钱包主页，展示总余额、可用余额、冻结余额三种状态
- 基于 shadcn/ui 封装充值/提现申请表单，使用 Zod 进行金额校验
- 使用 Recharts 实现余额可视化（堆叠条形图、径向条形图）

**后端**：
- 使用 Hono.js 开发钱包 RESTful API：余额查询、充值/提现申请、管理员审批
- 实现审批流程的事务处理：审批通过时同一事务中更新状态并修改余额

**数据库**：
- 设计钱包表：使用 `numeric(15,2)` 精确存储金额，区分三种余额状态
- 设计申请表：使用 PostgreSQL 枚举管理状态，CHECK 约束确保金额大于 0

**重点难点**：
- 金额精度：使用 numeric(15,2) 而非浮点数，前端格式化为两位小数
- 并发安全：数据库事务 + SQL 表达式更新，避免余额超扣
- 状态流转：设计状态机（pending → processing → completed/rejected），防止非法转换

#### 版本管理模块（多平台版本发布、强制更新）

**前端**：
- 实现版本检测逻辑：应用启动时自动检查更新，比较版本号判断是否需要更新
- 封装更新弹窗组件：显示版本信息、更新描述、下载进度条
- 使用 Tauri 插件实现跨平台下载安装（Android APK、Windows EXE、macOS DMG）

**后端**：
- 开发版本管理 API：获取版本列表、按平台筛选最新版本

**数据库**：
- 设计版本表：包含版本号、平台、下载链接、是否强制更新等字段
- 使用联合唯一约束（version + platform）防止重复版本

**重点难点**：
- 版本号比较：实现语义化版本比较函数，正确处理 1.0.10 > 1.0.9 的情况
- 强制更新：检测到强制更新时阻止用户关闭弹窗，必须更新后才能使用
- 流式下载：使用流式下载 + 进度回调，减少内存占用

#### 成员树状显示模块（层级成员管理）

**前端**：
- 使用 useMemo 将扁平数据构建为树形结构，根据 path 字段建立父子关系
- 实现展开/收起功能：使用 useState 管理展开状态，递归渲染子节点
- 使用 CSS paddingLeft 实现视觉缩进，层级越深缩进越大
- 支持搜索、状态筛选、层级筛选功能

**后端**：
- 使用 PostgreSQL ltree 扩展存储层级路径（如 root.child1.child2）
- 使用 ltree 的 `<@` 操作符查询当前用户及其所有下级成员

**数据库**：
- 成员表使用 ltree 类型的 path 字段存储层级路径
- 基于 path 实现权限控制：用户只能查看自己和下级成员

**重点难点**：
- 树形结构构建：动态判断根节点（父节点不在数据集中则为根节点）
- ltree 查询：使用 `path <@ 'root.child'::ltree` 查询所有子孙节点
- 渲染性能：使用 useMemo 缓存计算结果，仅数据变化时重新构建

**开发流程**：
1. 需求分析：根据原型图梳理功能点和页面交互
2. 数据库设计：设计表结构、字段类型、约束和索引
3. API 开发：定义接口规范（OpenAPI），实现 Handler 和业务逻辑
4. 前端开发：搭建页面结构，对接 API，实现交互效果
5. 联调测试：前后端联调，测试边界情况

### Huivo Admin - 客户与玩法管理后台
*个人项目 | 约 34,000 行代码*

**技术栈**：React 19, Vite 7, TanStack Router, Hono.js, Bun, PostgreSQL, Drizzle ORM, React Hook Form, pinyin-pro

**负责模块**：客户管理、玩法配置、产品管理（与 EzLottery 双数据库联动）

**前端**：
- 使用 React Hook Form + Zod 构建客户表单，支持多店铺（4 个店铺）、多来源（国际站/国内站/线下）选择
- 实现客户列表页面：支持按店铺、来源、状态筛选，支持客户名称搜索
- 构建玩法配置页面：支持奖品等级的动态增删改，实时计算奖品总数和总金额
- 开发产品管理模块：产品列表按客户隔离，支持关联玩法模板

**后端**：
- 开发客户 CRUD 接口：创建客户时同步到 EzLottery 系统（创建 user、organization、member、account 四张表）
- 实现玩法模板机制：玩法作为奖品配置模板，产品关联玩法时自动复制奖品配置
- 设计双数据库连接：huivo-admin 主库 + ezlottery 参考库，通过 `customerOrgId` 关联客户数据
- 实现权限隔离：用户只能查看和管理分配给自己的客户及其产品

**数据库**：
- 设计客户表：包含客户编号、店铺、来源、负责人等字段，建立多个索引优化查询性能
- 设计玩法表和奖品表：使用 `productId` 区分模板奖品和产品关联奖品
- 跨数据库关联：通过 `customerOrgId` 字段关联 ezlottery.organization 表

**开发流程**：
1. 原型评审：与产品确认页面布局和交互细节
2. 接口设计：编写 OpenAPI 规范，定义请求/响应类型
3. 数据库设计：设计表结构和索引，考虑跨系统关联
4. 后端开发：实现 API Handler，处理跨数据库事务
5. 前端开发：搭建页面，对接 API，实现表单校验
6. 集成测试：测试跨系统数据同步的正确性

**重点难点**：
- 拼音搜索：使用 pinyin-pro 库将客户名称转换为拼音 slug，存储到数据库支持模糊查询
- 跨系统同步：创建客户时需要在 ezlottery 数据库中创建 4 张表的记录，使用事务确保原子性
- 双数据库操作：产品数据存储在 ezlottery 库，玩法关联存储在 huivo 库，使用 Promise.all 并行查询
